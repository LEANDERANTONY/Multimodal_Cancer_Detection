# ADR-002 - Hybrid Notebook-Plus-Modules Repository Structure
**Date:** 2026-03-27  
**Status:** Accepted  
**Author:** Leander Antony A  

---

## Context

The project evolved through a notebook-first research workflow with several competing realities:

- the latest experimental logic and thesis-ready outputs lived in a large end-to-end notebook
- the cleaner GitHub-tracked repository already had a more maintainable modular scaffold under `src/`, `configs/`, and `docs/`
- raw data, processed CT images, embeddings, model checkpoints, and thesis assets were too large or too local to belong in Git

After consolidation, the repository needed to support both of these truths at once:

- preserve the latest research notebook and curated outputs
- continue moving stable logic into importable modules without breaking the active workflow

---

## Decision

The repository will use a hybrid structure:

### 1. Keep one primary notebook as the active research orchestration surface

- `notebooks/01_multimodal_cancer_detection.ipynb` remains the main end-to-end experiment narrative
- it is allowed to own exploratory analysis and presentation logic

### 2. Move stable reusable logic into `src/`

Shared logic should live in modules for:

- path and reproducibility helpers
- CT preprocessing and analysis
- embeddings and clustering
- CT model training and evaluation
- biomarker modelling
- fusion utilities
- summary and report assembly

### 3. Keep tracked artifacts lightweight and curated

Tracked:

- code
- configuration
- documentation
- curated reports
- curated figures

Ignored:

- raw data
- processed data
- embeddings
- model checkpoints
- thesis working materials

### 4. Use `uv` as the environment and dependency source of truth

The maintained dependency workflow is:

- `pyproject.toml`
- `uv.lock`
- `.python-version`

`requirements.txt` is not part of the active maintained path.

---

## Consequences

**Positive:**

- keeps the latest research workflow usable without waiting for a full rewrite
- makes stable code easier to reuse, test, and document
- keeps the Git repository practical and lightweight
- reduces the risk of losing current thesis outputs during cleanup

**Negative:**

- the notebook still remains a large and important runtime surface
- some exploratory logic continues to live outside `src/`
- reproducibility is improved but not yet fully automated end to end

---

## Future Work

1. Continue extracting notebook-only helper logic into `src/`
2. Add lightweight automated tests around stable module boundaries
3. Introduce thin scripts or CLIs for common tasks once the reusable surface settles
4. Keep documentation aligned with the actual hybrid state of the repository
